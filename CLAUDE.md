# Grooz iOS — Project Rules

Conventions Claude must follow when working in this repo.

## Git flow

This project uses **GitFlow**. Never commit directly to `main` or `develop`
(both are protected).

| Branch | Base | Merges into | Purpose |
| ------ | ---- | ----------- | ------- |
| `main` | — | — | Production. Every commit is a released, tagged version. |
| `develop` | `main` | — | Integration for the next release. Always one minor ahead. |
| `feature/*` | `develop` | `develop` | New work. |
| `release/*` | `develop` | `main` **and** `develop` | Stabilize a release (cut weekly). |
| `hotfix/*` | `main` | `main` **and** `develop` | Urgent production fix. |

- Feature work → branch from `develop`, PR back into `develop`.
- Releases → cut `release/X.Y.0` from `develop`, then immediately bump `develop`
  to the next minor. The release branch triggers a TestFlight build.
- Hotfixes → branch from `main` as `hotfix/X.Y.Z`, PR into `main`; also merge to
  `develop` keeping develop's higher version.

## Branch naming

`<type>/<short-kebab-description>` — lowercase, hyphenated, no spaces.

- `feature/user-profile-screen`
- `feature/grdb-caching`
- `release/1.4.0` — release branches are named by their exact `X.Y.0` version.
- `hotfix/1.4.1` — hotfix branches by their `X.Y.Z` patch version.

Allowed types: `feature`, `release`, `hotfix`. All non-release/hotfix work
(including chores, CI, docs) uses `feature/*`.

## Commit messages — Conventional Commits

Format: `type(scope): subject`

- **type**: `feat`, `fix`, `chore`, `ci`, `docs`, `refactor`, `test`, `build`, `perf`, `style`
- **scope** (optional): the area touched, e.g. `signing`, `ci`, `users`, `tuist`, `deps`
- **subject**: imperative mood, lowercase start, no trailing period, ≤ 72 chars

Body (optional): explain **what and why**, wrapped ~72 cols, after a blank line.

Examples:
- `feat(users): persist fetched users with GRDB`
- `fix(signing): use correct match profile name for release`
- `ci(testflight): trigger deploy on release branches`
- `chore(deps): bump Alamofire to 5.12.0`

Every commit message ends with the trailer:
```
Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
```

## Pull requests

Because merges are **squash-only**, the PR title becomes the squash commit
message, and (for `release/*`) feeds the TestFlight "What to Test" changelog.
So PR titles and descriptions are release notes — write them accordingly.

**PR title** — same Conventional Commit format as commits:
`type(scope): subject` (e.g. `feat(users): add users list screen`).

**PR base branch**:
- `feature/*` branches → target **`develop`**
- `release/*` and `hotfix/*` → target **`main`** (and merge back to `develop`)

**PR description** — follow this template. Adapt as needed (a trivial change
can be terse), but cover these points:

```
## What
<One-paragraph, user-facing summary. This is the release note — for release/*
 PRs it is what testers read in TestFlight, so describe behavior, not files.>

## Why
<The problem or reason for the change.>

## How to verify
<Concrete evidence, not "should work": test results (e.g. "9 tests pass"),
 "BUILD SUCCEEDED", steps to reproduce/exercise, screenshots.>

## Follow-ups
<Required secrets/env/migrations/manual steps, or "None".>
```

Guidelines:
- The **What** section is the changelog — write it as behavior a user/tester
  would recognize, never "refactored X.swift".
- **How to verify** must cite real evidence (a build, tests run), not intent.
- Link the issue if one exists, e.g. `Closes #12`.

## Versioning

- **Marketing version** (`MARKETING_VERSION`) lives in `Version.xcconfig` — the
  single source of truth. Bump it when cutting a `release/*` branch.
- **Build number** (`CURRENT_PROJECT_VERSION`) comes from CI via
  `TUIST_CURRENT_PROJECT_VERSION` (`github.run_number`) — never hardcode it.
- Tag `main` `vX.Y.Z` on each release.

## Project specifics

- Xcode project is generated by **Tuist** — never edit `*.xcodeproj` /
  `*.xcworkspace` (gitignored). Change `Project.swift` / `Tuist/Package.swift`,
  then `tuist install && tuist generate`.
- fastlane config is **env-driven** (`fastlane/.env.default` committed,
  `fastlane/.env` secret) — don't add Gemfile/Appfile/Matchfile back.
- Verify changes with a real build/test (`fastlane test`, `xcodebuild build`),
  not just the SourceKit indexer (it reports false "No such module" errors for
  Tuist-resolved packages).
